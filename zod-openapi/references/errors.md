# RFC 9457 Error Responses

## Why RFC 9457
Structured error contracts beat HTML pages and bespoke `{ error: string }` blobs. Problem Details gives every HTTP client a predictable base shape, while extension members carry the operational data your platform actually needs.
For agents, this matters even more: they can route, retry, escalate, and correlate deterministically instead of scraping prose. One format for every 4xx/5xx kills branching logic and stops error handling from degrading into folklore.

## Canonical Shape
Default template for every error response:

```json
{
  "type": "https://api.example.com/errors/rate-limited",
  "title": "Error 1015: You are being rate limited",
  "status": 429,
  "detail": "You are being rate-limited by the website owner's configuration.",
  "instance": "9d99a4434fz2d168",
  "error_code": 1015,
  "error_name": "rate_limited",
  "error_category": "rate_limit",
  "ray_id": "9d99a4434fz2d168",
  "timestamp": "2026-03-09T11:11:55Z",
  "zone": "<YOUR_DOMAIN>",
  "cloudflare_error": true,
  "retryable": true,
  "retry_after": 30,
  "owner_action_required": false,
  "what_you_should_do": "**Wait and retry.** This block is transient. Wait at least 30 seconds, then retry with exponential backoff.",
  "footer": "This error was generated by Cloudflare on behalf of the website owner."
}
```

## Required Members (RFC 9457)
| Member | Type | Description | Rules |
|---|---|---|---|
| `type` | string (URI) | Stable identifier for the error class | MUST resolve to real docs; dead links are contract breakage |
| `title` | string | Short summary of the error class | STATIC per type; occurrence-specific context goes in `detail` |
| `status` | integer | HTTP status code | MUST match the actual response status |
| `detail` | string | Human-readable description of this occurrence | Dynamic, specific, debuggable |
| `instance` | string | Unique identifier for this occurrence | Use request ID, ray ID, or trace ID |

## Extension Members
| Member | Type | Purpose | Rules |
|---|---|---|---|
| `error_code` | integer | Stable machine code | Register centrally; never rename or reuse |
| `error_name` | string | Stable machine name | snake_case; never drift from docs |
| `error_category` | string | Coarse routing bucket | Keep taxonomy small and operational |
| `ray_id` | string | Correlation identifier | Usually equal to `instance` |
| `timestamp` | string | Occurrence time | ISO 8601 UTC |
| `zone` | string | Domain or tenant context | Include tenant/domain scope when it matters |
| `cloudflare_error` | boolean | Origin flag | For non-Cloudflare APIs, rename to `{service}_error` or omit |
| `retryable` | boolean | Retry signal | Drives client automation directly |
| `retry_after` | integer | Retry delay in seconds | Present when `retryable: true` |
| `owner_action_required` | boolean | Escalation signal | `true` means stop retrying and escalate to operator/owner |
| `what_you_should_do` | string | Action guidance | Markdown allowed; keep it imperative and specific |
| `footer` | string | Attribution/context string | Useful for platform-generated errors |

## Content-Type Rules
- Use `application/problem+json; charset=utf-8` for every 4xx and 5xx response.
- If the client sends `Accept: application/problem+json`, return that media type explicitly.
- If the client sends `Accept: application/json`, return the identical body as `application/json; charset=utf-8`.
- For `429` and `503`, set the `Retry-After` HTTP header in addition to the `retry_after` body field.

## Error Category Taxonomy
### Cloudflare 1xxx categories
| Category | What it means | Agent action |
|---|---|---|
| `access_denied` | Platform blocked access | Stop, inspect policy, escalate if needed |
| `rate_limit` | Temporary throttling | Back off using `retry_after` |
| `dns` | Upstream resolution failure | Retry if transient; escalate if persistent |
| `config` | Owner misconfiguration | Stop and notify operator |
| `tls` | Handshake or certificate failure | Stop and escalate to infrastructure owner |
| `legal` | Legal/policy restriction | Do not retry; escalate |
| `worker` | Worker/runtime execution failure | Log `instance`, escalate, maybe retry if marked |
| `rewrite` | Rewrite rule failure | Escalate to edge config owner |
| `snippet` | Snippet/runtime config failure | Escalate to deploy owner |
| `unsupported` | Feature not supported in current mode | Stop and choose another path |

### Generic API categories
| Category | What it means | Agent action |
|---|---|---|
| `validation` | Request contract violated | Fix fields and resubmit |
| `auth` | Authentication or authorization failed | Refresh credentials or escalate |
| `not_found` | Resource does not exist | Stop or create prerequisite resource |
| `conflict` | Current state rejects the change | Re-fetch state, reconcile, retry if appropriate |
| `rate_limit` | Server asked you to slow down | Back off and retry later |
| `server` | Platform failed internally | Log, correlate, escalate |

## Status Code → Category Mapping
| Status | Category | Notes |
|---|---|---|
| `400` | `validation` | Malformed request or unsupported combination |
| `401` | `auth` | Missing or invalid credentials |
| `403` | `auth` | Authenticated but not permitted |
| `404` | `not_found` | Missing resource or wrong identifier |
| `409` | `conflict` | Concurrent update, duplicate, invalid state transition |
| `422` | `validation` | Structurally valid request, semantically invalid fields |
| `429` | `rate_limit` | Must include `retry_after` and `Retry-After` |
| `500` | `server` | Never leak internals |
| `503` | `server` | Temporary unavailability; may be retryable |

## Agent Consumption Flow
```text
1. Parse JSON (Content-Type guarantees structure).
2. Read `retryable`:
   - true  → sleep `retry_after`, exponential backoff.
   - false → read `what_you_should_do` and `owner_action_required`.
3. Read `error_category` to route:
   - auth       → refresh credentials / escalate.
   - validation → fix using `errors` array.
   - rate_limit → backoff + retry.
   - server     → log + escalate.
4. Use `instance` / `ray_id` for correlation in logs and support.
```

## Error Catalog Rules
- Register every `error_code` and `error_name` in one central catalog.
- Never rename an existing code, never reuse a retired code.
- Every `type` URI MUST resolve to real documentation.
- `title` is static per error class; dynamic context belongs in `detail`.
- `422` validation failures SHOULD include `errors: Array<{ field, message, code }>`.

## TypeScript Shape
```typescript
/** RFC 9457 Problem Details with Cloudflare-style operational extensions. */
export interface ProblemDetails {
  /** Stable URI for this error class. */
  type: string;
  /** Static summary for the error class. */
  title: string;
  /** HTTP status code; must equal the actual response status. */
  status: number;
  /** Occurrence-specific explanation. */
  detail: string;
  /** Unique identifier for this occurrence. */
  instance: string;
  /** Stable numeric machine code. */
  error_code?: number;
  /** Stable snake_case machine name. */
  error_name?: string;
  /** Routing category such as validation, auth, rate_limit, or server. */
  error_category?: string;
  /** Correlation identifier, often equal to instance. */
  ray_id?: string;
  /** ISO 8601 timestamp for the occurrence. */
  timestamp?: string;
  /** Domain, tenant, or zone context. */
  zone?: string;
  /** Platform-origin flag; rename per service or omit outside Cloudflare. */
  cloudflare_error?: boolean;
  /** Whether automated retry is appropriate. */
  retryable?: boolean;
  /** Retry delay in seconds when retryable is true. */
  retry_after?: number;
  /** Whether a human owner/operator must intervene. */
  owner_action_required?: boolean;
  /** Action guidance for humans and agents. */
  what_you_should_do?: string;
  /** Attribution or extra context string. */
  footer?: string;
}

export interface ValidationProblemDetails extends ProblemDetails {
  errors: Array<{ field: string; message: string; code: string }>;
}
```

## Modeling ProblemDetails with zod-openapi

```ts
import * as z from 'zod/v4';

const ProblemDetails = z.object({
  type: z.string().url().meta({ example: 'https://api.example.com/problems/not-found' }),
  title: z.string(),
  status: z.number().int(),
  detail: z.string().optional(),
  instance: z.string().optional(),
}).catchall(z.unknown()).meta({ id: 'ProblemDetails' });
```

```ts
responses: {
  '404': {
    description: 'Not Found',
    content: {
      'application/problem+json': {
        schema: ProblemDetails,
      },
    },
  },
}
```

Register `ProblemDetails` once with `.meta({ id: 'ProblemDetails' })` and reuse it from every failure-mode response so the emitted OpenAPI document carries one canonical component. That keeps the contract stable and stops error payload drift across endpoints.

`title` stays static per error class and occurrence-specific context goes in `detail`, which is the RFC 9457 split clients can depend on. `.catchall(z.unknown())` is how the Zod schema preserves RFC 9457 extension members without over-constraining the shape.
